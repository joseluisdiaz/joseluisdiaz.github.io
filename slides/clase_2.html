Tabla de Hechos

  • Los hechos representan medidas del negocio. Cada fila representa un evento medido en cierta granularidad
  • La idea de que cada evento en el mundo físico tiene una relación uno-a-uno con las tablas de hechos es fundacional en términos del modelo dimensional.
  • Los hechos más útiles generalmente son numéricos y aditivos
  • Las medidas textuales generalmente es la descripción de algo más, y termina siendo una dimensión.
  • Usualmente las PK de la tabla de hechos es compuesta por varias FK de las tablas de dimensiones

Tabla de dimensiones

  • Describen: quién, qué, dónde, cuándo, cómo, por qué
  • No es poco común que las tablas de dimensiones tengan de 50 a 100 columnas. Generalmente tienen menos cardinalidad que las tablas de hechos
  • Juegan un rol fundamental: Son las restricciones por las cuales se van a generar todos los reportes
  • Hay un dilema constante: si un valor numérico es un Hecho o una Dimensión
  • Las tablas de dimensiones usualmente representan una relación de jerarquía
  • Altamente de-normalizada

Drill down

Examinando data agregada, partiéndolo por atributo.

        Select state, brand, Sum(qty*price)
        From Sales F, Store S, Item I
        Where F.storeId = S.storeId And F.itemID = I.itemId
        Group By state, brand

--> Drill Down 

        Select state, category, brand, Sum(qty*price)
        From Sales F, Store S, Item I
        Where F.storeId = S.storeId And F.itemID = I.itemId
        Group By state, category, brand

Roll Up

Examinando data agregada, partiéndolo por atributo.

ETL

34 subsistemas

  • La extracción. La recopilación de datos en crudo de los sistemas de origen y por lo general, la escritura en el disco del entorno del ETL antes de que cualquier reestructuración significativa de la los datos se lleva a cabo. Subsistemas del 1 al 3 de apoyo al proceso de extracción
  • Limpieza y ajustado. El envío de los datos crudos a través de una serie de procesos del ETL para mejorar la calidad de los datos, la fusión de los datos de dos o más fuentes, ajustar las dimensiones y las métricas. Los subsistemas de 4 a 8 esta encargados de esta etapa.
  • La entrega. Estructuración física y la carga en los modelos dimensionales en el destino. Subsistemas de 9 a 21 proporcionan la capacidades para la entrega de los datos para el servidor.
  • Gestión. La gestión de los sistemas y procesos relacionados al medio ambiente del ETL de una manera coherente. Subsistemas de 22 a 34 describen los componentes necesarios para apoyar la gestión en las diferentes etapas del sistema de ETL.

ETL - Extracción

Subsistema 1: Data Profiling

    Data profiling, es el análisis técnico de los datos para describir su contenido, consistencia, y estructura. En cierto sentido, toda vez que se realiza una consulta SELECT DISTINCT en una en cierto modo se esta haciendo data profiling. Provee al equipo con una orientacion en cuanto a la cantidad de datos que van a ser necesarios limpiar, y ser posteriormente ajustados.

ETL - Extracción

Subsistema 2: Change Data Capture System (Delta)

  • Simplemente es un sistema que transifiere los datos que han cambiado desde la última carga
  • Aislar la fuente de datos modificados para permitir el procesamiento selectivo en lugar de una actualización completa.
  • Captura todos los cambios (DELETE, UPDATE, INSERT) introducida en los datos de origen, incluidos los cambios realizados a través de interfaces no estándar.
  • Marcar los cambios de los datos datos con los códigos para distinguir las correcciones de errores de actualizaciones.

ETL - Extracción

Subsistema 3: Extract System

  • La situación más común es que cada fuente puede estar en un un medio ambiente.
  • Hay dos metodos para extraer datos, desde archivos y desde un flujo (stream)

ETL - Limpieza y ajustado

  • 4: Data cleasing system el proceso de limpieza de datos ETL se utiliza para corregir los datos sucios, capturar la calidad y medirla. Quality Screens Funcionan como filtros que realizan test contra el flujo de información
  • 5: Error Event Schema es un esquema dimensional centralizado cuyo objetivo es registrar cada evento de error producido por una Quality Screens en cualquier parte del pipeline del ETL.
  • 6: Audit Dimension Assembler. Audit Dimesion es una dimensión que se genera por el ETL por cada tabla de hechos y matiene un registro de metadatos sobre cada row.
  • 7: Deduplication System. En un entorno donde hay muchas fuentes de origen es fundamental poder detectar duplicados.
  • 8: Conforming System. Es el responsable de consolidar y distribuir las dimensiones

ETL - Prepararlo para la presentación

  • 9: Slowly Changing Dimension Manager. Debe determinar como proceder con los cambios de las dimensiones. Cuando una dimensión cambia, hay tres respuestas básicas, 'sobreescribir', agregar una 'fila' o una 'columna'
  • 10: Surrogate Key Generator. Es importante poder definir claves para las dimensiones en el proceso de ETL.
  • 11: Hierarchy Manager. Se utiliza para manejar las jerarquías tantos de-normalizadas como las normalizadas (Snowflakes)
    • )
  • 12: Special Dimensions Manager. Es donde se manejan las dimensiones que requieran un tratamiento especial dentro del ETL, como por ejemplo Date/Time, o dimesiones manejadas por el usuario.

ETL - Prepararlo para la presentación

Subsistema 13: Fact Table Builders

Se encarga de construir los tres tipos primarios de tabla de hechos

  • Transaction FTL. La granularidad de un evento medido en un instante. Por ejemplo cuando una cajera de supermercado pasa un ítem por el Scanner.
  • Periodic Snapshot FTL. La granularidad esta representa un conjunto de medidas, por ejemplo el saldo de una cuenta bancaria mensual.
  • Accumulating Snapshot FTL. Representa el estado actual de un evento que cambia y que tiene un principio y un final finito. Cualquier proceso que pueda ser representado con una maquina de estados finito es un buen ejemplo

ETL - Prepararlo para la presentación

  • 14: Surrogate Key Pipeline . Está encargado de reemplazar las NaturalKeys por claves suplentes.
  • 15: Multivalued Dimension Bridge Table Builder. Cuando es necesario acceder a valores de otra granularidad, para acceder a este conjunto de valores se hace a través de una tabla bridge.
  • 16: Late Arriving Data Handler. Cuando los hechos y las dimensiones no llegan al mismo tiempo este subsistema se encarga de manejar este caso.
  • 17: Dimension Manager System. Es el subsistema que centraliza y prepara la publicación de las dimensiones.
  • 18: Fact Provider System. Recibe las dimensiones, y se encarga de administrar las tabla de hechos.
  • Los subsistemas 19,20 y 21 son respectivamente: Aggregate Builder, OLAP Cube Builder, Data Propagation

ETL - Manejando el entorno del ETL

Todo el proceso de ETL no tendrá sentido si no se lo administra de manera adecuada, para eso es importante destacar estos criterios

  • Confiabilidad. Los procesos ETL se deben ejecutar de forma coherente, deben correr y completarse.
  • Disponibilidad. El almacén de datos debe cumplir con sus acuerdos de nivel de servicio (SLAs). El almacén deberá estar listo y disponible según lo prometido.
  • Administrabilidad. Un almacén de datos con éxito nunca termina. Constantemente crece y cambia al mismo tiempo que cambia negocio. Los procesos ETL necesitan con evolucionar también.

Los subsistemas que van del 22 al 34 son los siguientes: Job Scheduler, Backup System , Recovery and Restart System, Version Control System, Version Migration System, Work flow Monitor, Sorting System , Lineage and Dependency Analyzer, Problem Escalation System, Parallelizing/Pipelining System , Security System , Compliance Manager, Metadata Repository Manager.
